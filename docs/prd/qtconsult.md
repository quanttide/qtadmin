# 量潮咨询 PRD

> 量潮科技旗下业务线之一。

## 业务模型

咨询服务的核心工作流只有一件事：**梳理客户情况 → 制定咨询策略**，这个循环贯穿整个咨询周期。

```
信息看板（客户是什么情况） ←→ 策略看板（我们怎么应对）
     ↑                             ↑
     持续更新，贯穿全程不退场
```

两个看板的关系：
- **信息看板**回答：客户现在的真实状况是什么（画像、发现、沟通记录）
- **策略看板**回答：基于当前认知，我们打算怎么做（诉求、切入路径、决策链路）
- 一条新发现 → 策略需要重新审视。如果发现了什么但不调整策略，这条发现就不值得记

## 场景一：接洽期 — 判断要不要接，怎么切入

**决策频率**：每月数次。**决策者**：咨询业务负责人。

### 决策现场

信息看板：

1. **客户画像** — 行业、规模、数字化成熟度（初次接触只能给出初步判断，随着沟通深入持续修正）
2. **发现清单** — 初次沟通产出的初步观察（客户在解决什么问题、谁在推动、阻力在哪）
3. **决策链初探** — 谁是这个项目的关键推动者，决策链上还有谁没接触到

策略看板：

1. **战略诉求判断** — 客户说的需求 vs 我们判断的真实需求（往往不是一回事）
2. **切入策略草案** — 如果接，从哪个痛点切入、分几步走、第一步做什么
3. **接/不接建议** — 团队能力能否覆盖、战略价值如何、产能是否允许

点"接"后，系统复制当前两个看板的内容进入项目空间，作为正式咨询的起点。不需要重新填表。

### 替代路径

系统积累足够多的客户项目后，在新客户接洽时自动匹配历史项目画像，提示相似度、历史结果和可复用的行业认知。

## 场景二：方案期 — 认知深化驱动策略迭代

**决策频率**：每次客户接触后。**决策者**：项目顾问。

### 决策现场

每次与客户接触后，顾问更新两样东西：

信息看板更新：

1. **发现清单新增** — 这次沟通发现了什么之前不知道的（业务痛点、组织阻力、竞争对手）
2. **决策链更新** — 谁的态度变了、谁是新出现的决策者、谁实际没有决策权
3. **沟通记录** — 纪要归档，供后续查证

策略看板随之更新：

1. **战略诉求校准** — 初始判断是否准确，客户的真实诉求是否浮现
2. **切入策略调整** — 新发现是否影响路径设计（例如发现中层抗拒，则需在策略中增加中层动员步骤）
3. **决策链路刷新** — 每个人的立场更新，应对策略是否需要调整

系统不要求顾问"记录项目进度"，只要求顾问回答"这次接触后，我们对客户的认知有什么变化"——这个变化自动驱动策略更新。

### 替代路径

系统识别出某类发现反复出现（如"客户IT能力不足"在多个项目中都是关键制约因素），在后续项目中提前在策略模板中提示风险点。

## 场景三：交付期 — 执行中的认知校准

**决策频率**：按交付节奏。**决策者**：项目顾问。

### 决策现场

信息看板持续补充执行中发现的新情况（方案落地中的实际反馈、客户组织的真实反应）。

策略看板从"计划做什么"变为"实际在做什么"，跟踪：
- 每一步的实际进展 vs 预期
- 需要调整的环节及原因
- 新发现触发的策略修正记录

系统不要求填进度百分比。顾问只做两件事：记录新发现、更新策略。进度是策略执行的副产物。

### 替代路径

当策略修正频率过高（如一周内修改超过3次）时，系统提示"当前策略可能需要重新审视"，触发复盘。

## 场景四：复盘期 — 认知资产化

**决策频率**：每个项目结束后。**决策者**：咨询业务负责人。

### 决策现场

两个看板完整保留，系统在项目关闭时自动整理：

1. **发现清单 → 行业认知**：项目过程中对客户行业的理解，哪些是可复用的
2. **策略迭代轨迹 → 方法论**：策略修正了哪些次、因为什么修正、修正后效果如何
3. **决策链路演变 → 客户关系模型**：不同角色的态度变化规律，谁在什么条件下转变

负责人只需要标注：哪些认知和方法论值得保留，哪些是这家客户特有的、没有复用价值。

### 替代路径

新项目启动时，系统自动关联可复用的发现和方法论："上一个制造业项目的切入策略是X，其中Y环节被验证有效，建议参考。"

## 与全景图的关系

量潮咨询在全景图中只展示两样东西：
- 进行中项目的**当前阻塞点**（卡在客户还是卡在自己）
- 待承接客户的**匹配度评估**（团队能力能否覆盖）

所有深度工作（信息看板、策略看板）在咨询详情页完成，不涌入全景图。
