# IT咨询服务业务需求说明书

## 服务定位

为企业客户提供IT咨询，核心目标是以技术手段驱动企业完成数字化转型与智能化转型。咨询团队需要将自身的技术能力与客户的战略诉求进行精准匹配，而非推销标准化的技术方案。

## 业务模型

咨询服务的核心工作流是一条闭环：

```
梳理客户情况 → 制定咨询策略 → 执行 → 发现新情况 → 调整策略 → 继续执行
```

这个循环贯穿整个咨询周期，从接洽期到交付完成。关键在于：**对客户的认知和咨询策略之间必须是双向驱动的**——认知变了策略就得变，策略执行中也会发现新的认知。

现实中最容易出问题的地方恰恰在这里：认知和策略脱节。顾问知道了很多客户信息，但没有把这些信息转化为策略调整；或者策略定了就不再回头审视，即使客户的情况已经变了。

## 核心难点

### 客户信息难以结构化

与客户的每次接触都会产出大量信息：业务痛点、组织关系、决策链、竞争态势、文化特征。这些信息目前散落在沟通纪要、聊天记录和个人笔记中，下次要制定策略时找不到，或者只有接触过的人才知道。

### 认知到策略的转化断裂

知道"客户中层抗拒变革"和"知道该怎么做"之间有一条鸿沟。顾问往往停留在"知道了"的阶段，没有系统性地把每条认知转化为对应的策略动作。结果是方案设计依赖个人经验，换一个人就得重新摸索。

### 全程一致性难保持

接洽期对客户的判断、方案期的深入理解、交付期的执行反馈——这三个阶段的认知应该是递进校准的关系，但实际上经常前后矛盾：方案期的发现推翻了接洽期的假设，但策略没有跟着调整。

## 系统价值

系统要解决的不是"帮顾问记录客户信息"，而是**保证每一条认知都对应一条策略**，并且这种对应关系在整个咨询周期中持续维护。

具体来说：
- **接洽期**：用有限的初步信息判断要不要接、怎么切入
- **方案期**：每次接触后的新发现自动触发策略审视
- **交付期**：执行中的实际反馈反向修正策略
- **复盘期**：完整的认知-策略演变轨迹沉淀为可复用的行业方法论

## 长期策略

每完成一个项目，团队对某个行业的认知和理解就加深一层。长期需要建立：

- **认知资产化**：项目过程中对客户行业的理解，哪些是这家客户特有的、哪些是行业通用的
- **策略模式识别**：不同情境下什么策略有效、什么策略无效，识别重复模式
- **行业知识库**：通过项目积累形成行业级的认知地图和策略库，新项目启动时不再从零开始
