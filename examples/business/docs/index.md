# 商务拓展职能成熟度实验报告

Assets → Data → Code → Docs 管线全流程输出。

## 管线回顾

```
Assets(4份) → Data(schema+rules+samples) → Code(calculator+tests) → Docs(report)
```

## 做了什么

1. **Assets**：从章程、手册、教程、档案中提取报价相关规则，整理为结构化的资产文件
2. **Data**：设计报价业务实体 schema，标注 15 条规则的可编码性，用教程案例作为样本数据
3. **Code**：用 Python 实现报价计算引擎，8 个测试全部通过
4. **Docs**：产出可编码性评分表和编码模糊点清单

## 可编码性评分

评分标准：5=直接可编码，4=微调后可编码，3=需补充信息，2=模糊需重写，1=无法编码。

## 模糊点清单

### 章程层面
- 第十八条"金额标准"与"让利比例"由管理层另行制定——具体数值未定义，编码时视为配置参数
- 第十四条"其他类型客户"一事一议——无定价规则
- 资深/中级/初级人员定价待补充

### 手册层面
- 人员等级表中"资深""中级""初级"的对应人选和服务对象均为"待补充"
- 谈判话术模板中让利规则引用报价档案，但未说明保本线的计算方式

### 教程层面
- 无明确金额数字，只有定价思维框架
- "成本大约 X 元"中的 X 未给出，标注"按各业务线核算"

## 编码发现的 5 个问题

**1. "保本线"未定义**
- 出处：profile 让利规则
- 问题：首次合作不低于保本线，但保本线的计算方式和具体数值未定义
- 建议：补充保本线的计算公式或默认值（如 = 成本 × 1.2）

**2. 资深/中级/初级人员定价缺失**
- 出处：profile 人员分级
- 问题：五个等级中只有首席和高级有具体价格，其余标注为"待补充"
- 建议：补充缺失的人员定价，或明确其不可用状态

**3. 金额标准与让利比例未定义**
- 出处：bylaw 第十八条
- 问题：章程规定"金额标准"和"让利比例"由管理层另行制定，但当前无具体数值
- 建议：制定并补充这两个配置参数

**4. 高校客户定价依赖个案**
- 出处：profile 客户类型
- 问题：高校客户为"按合作协议"，无法编码为确定的定价规则
- 建议：如有常见合作模式，定义标准价格区间

**5. 稀缺性/创新溢价/长期价值评估无量化标准**
- 出处：bylaw 第十一至十三条
- 问题：三个评估要素均为定性判断，无法直接编码
- 建议：如需要这些评估自动化，可设计评分量表（如 1-5 分）将定性转为定量

## 值得记录的观察

- 教程的四层定价思维模型是好框架，但具体定价完全依赖报价档案
- 手册的 10 条"不得"规则非常适合编码为断言/校验
- 报价单 3:4:3 结构比例清晰，可直接编码为默认模板

## 核心结论

**可编码率 53%**：约一半规则已清晰到可直接翻译为代码，另一半有明确改进方向。

**文档质量度量**：可编码性评分提供了一个客观的文档质量度量——章程/手册/档案的质量 = 可编码性得分。5 分说明文档清晰，1-2 分说明有改进空间。

**治理即代码**：Assets → Data → Code → Docs 管线可行。结构化时发现保本线未定义，编码时发现人员定价缺失，评估时发现稀缺性无量化标准——这些问题在纯文本阅读时容易被忽略，但在编码时变得不可回避。
